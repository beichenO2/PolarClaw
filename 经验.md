# Agent 项目工程经验（给下一个 Agent 的执行手册）

目标读者：接手新项目或新切片的 Agent。
目标：让你在没有人类详细说明的情况下也能正确推进项目。
阅读方式：遇到对应场景时，直接跳到对应章节执行，不需要通读全文。

---

## 一、文档体系（项目外部记忆）

每个软件项目必须维护以下 5 个文件。缺少任何一个，你在多轮对话中将失去上下文。

```
docs/
  state.md        ← SSOT（单一事实来源），最权威
  interfaces.md   ← API 契约，代码必须严格遵守
  decisions.md    ← 决策日志，只能追加，不能改写历史
  roadmap.md      ← 待办 Backlog，可以过时，不作为权威
  docOps.md       ← 文档操作规范本身（本手册的元文档）
```

**权威优先级（发生冲突时）：**

```
state.md > interfaces.md > decisions.md > roadmap.md
```

## 二、每个文档的完整模板

### 2.1 state.md

```markdown
# Project State (SSOT)

Last Updated: YYYY-MM-DD (附注本次更新原因)
Source of Truth: state.md is authoritative.
Conflict Rule:
- If other docs conflict with state.md, follow state.md.
- Propose patches to bring other docs in sync. Do not "average" conflicts.

## Project
Name: <项目名称>
Goal (v1): <一句话描述 v1 目标>

## Current Slice
Slice: <当前切片状态，格式为 "SN DONE -> SN+1 NOT STARTED" 或 "SN IN PROGRESS">

- S0: <描述> [状态标签]
- S1: <描述> [状态标签]
- ...

状态标签枚举（只能用这四种）：
  [NOT STARTED]
  [code written YYYY-MM-DD; pending run-verification]
  [DONE - verified YYYY-MM-DD]
  [IN PROGRESS]

Non-goals (for SN/SN+1):
- No <禁止做的事1>
- No <禁止做的事2>

## Architecture (high level)
Layers:
- Entry: <入口层>
- Service: <服务层>
- Core: <核心层>
Later:
- <未来层>

Data flow (SN):
<Component A> -> <endpoint> -> <Component B> -> <response> -> <Component A>

## Tech choices
Frontend: <具体选型，若未决定写 TBD (候选A / 候选B)>
Backend: <具体选型>
<其他组件>: <具体选型>

## Repo Layout
- <目录1>/   (<说明实际包含的文件和职责>)
- <目录2>/   (<说明>)
- docs/

## Run Targets
Backend:
- Port: <端口>
- Command: <完整启动命令>
- Health: GET /health -> {"status":"ok"}
Frontend:
- Port: <端口>
- Command: <完整启动命令>
- Calls backend base URL: http://localhost:<端口>

## Acceptance Tests
S0:
- [x/空格] <验收项1>
- [x/空格] <验收项2>
S1:
- [ ] <验收项>
SN（下一个切片预占位）:
- [ ] <验收项>
```

**state.md 填写规则：**
- `Last Updated` 每次修改时必须更新。
- `Slice` 行必须反映当前最新状态，格式固定。
- `Tech choices` 中未确定的写 `TBD (候选列表)`，确定后去掉括号内容改为实际值。
- `Repo Layout` 写实际存在的文件，不写"planned"；规划中的目录单独用注释说明。
- `Run Targets` 写完整可复制粘贴的命令，不写伪代码。
- `Acceptance Tests` 只在用户或终端确认通过后才打 `[x]`，不得提前打勾。

### 2.2 interfaces.md

```markdown
# Interfaces (Contract)

Rule:
- Treat this file as the API contract.
- Do NOT change request/response shapes in code unless you first propose a change here.

## HTTP API

### <HTTP方法> <路径>
Purpose: <一句话描述，标注适用切片>

Request JSON:
{
  "<字段>": "<类型描述>"
}

Response <状态码> JSON:
{
  "<字段>": "<类型描述>"
}

Error response (>=400) JSON:
{
  "error": {
    "code": "string",
    "message": "string"
  }
}

## Internal Interfaces (Backend)

### <接口名>
- <方法签名>
Notes:
- <约束>

(<未来功能标注，格式: "(Reserved for future)" 或 "(SN+)">)
### <接口名> (SN+)
- <方法签名>
```

**interfaces.md 填写规则：**
- 每个字段类型用括号注明可选性：`"number (optional)"`。
- Error response 格式全局统一，所有端点共用同一个 error schema。
- 未来切片的接口用 `(SN+)` 标注，不写实现细节，只写签名。
- 修改此文件前必须先提案，等用户确认后才能动代码。

### 2.3 decisions.md

```markdown
# Decisions Log (Append-only)

Rules:
- Append new entries only. Do not rewrite old entries.
- Each decision should be small, explicit, and reversible where possible.

## YYYY-MM-DD
- D001: <决策描述>。<一句话理由，不超过 20 字>。
- D002: <决策描述>。<理由>。

## YYYY-MM-DD (SN)
- D00N: <决策描述>。<理由>。
```

**decisions.md 填写规则：**
- 只能追加新 `## 日期` 段落，绝不修改已有行。
- 每个 D-ID 全局唯一，从 D001 顺序递增。
- 触发条件：框架选型、端口/协议决定、安全策略、数据存储方案、API 设计取舍。
- 不触发条件：变量命名、文件位置调整、格式修改。
- 每条格式：`- D<NNN>: <是什么>（<理由，不超过20字>）。`

### 2.4 roadmap.md

```markdown
# Roadmap / Backlog

## Done
- S0: <描述> [verified YYYY-MM-DD]
- S1: <描述> [verified YYYY-MM-DD]

## Now
- S2: <当前切片描述>

## Next
- S3: <描述>
- S4: <描述>

## Later / Parking Lot
- <长期待办>
- <长期待办>
```

**roadmap.md 填写规则：**
- `Done` 区：切片验收通过后移入，标注日期。
- `Now` 区：只有一项，即当前正在做的切片。
- `Next` 区：按优先级排列，不包含已在 Done/Now 中的条目。
- `Later` 区：没有明确优先级的长期愿望，可以一直停在这里。
- 同一个切片不能同时出现在两个区（移动时从原区删除）。

### 2.5 docOps.md

```markdown
# DocOps Policy (Update Declaration)

This project uses 4 docs as the external project memory:
- docs/state.md (SSOT, authoritative)
- docs/interfaces.md (contract)
- docs/decisions.md (append-only log)
- docs/roadmap.md (backlog, allowed to drift)

## Authority & Conflict Rules
1) state.md is the Single Source of Truth (SSOT). If any conflict exists, follow state.md.
2) interfaces.md is a contract. Code MUST follow it. Do not change API shapes in code unless interfaces.md is updated first.
3) decisions.md is append-only. Never rewrite or delete historical entries; only append new decisions.
4) roadmap.md is a planning backlog. It may be outdated; do not treat it as authoritative.

## Update Triggers
（见第三章详细规则）

## Backup & Rotation (Rollback depth = 1)
（见第四章详细规则）

## Non-Fabrication Rule
The agent MUST NOT invent facts in docs.
If a required fact is unknown, write "TBD" and add a checklist item instead.

## Output Format Requirements
When asked to implement a slice/task, the agent must output:
1) A small plan (<= 6 steps)
2) Code changes (patch/diff or full files)
3) Test/verification steps (commands + expected results)
4) Doc patches for the 4 docs (if triggered by rules above)
```

## 三、文档更新触发规则

每轮任务结束时，对照下表判断哪些文档需要更新：

| 发生了什么 | 必须更新的文档 |
|-----------|--------------|
| 切片状态变化（code written / verified） | state.md |
| 框架/运行时选型确定 | state.md + decisions.md |
| 新增/删除/修改 API 端点或字段 | interfaces.md（需先提案）|
| 端口/启动命令变化 | state.md |
| 验收测试通过（用户或终端确认） | state.md（打 `[x]`）|
| 有新的框架/协议/安全取舍决定 | decisions.md |
| 切片验收通过，进入下一切片 | state.md + roadmap.md |
| 发现新功能需求 | roadmap.md |

**不触发更新的情况：**
- 变量重命名、代码格式调整
- 注释修改
- 测试文件内容变化
- 未经用户/终端确认的"应该能工作"

## 四、备份与轮换规则（Rollback depth = 1）

### 何时执行备份

每次准备修改任意一个文档之前，必须先备份全部 4 个文档。

### 执行命令（固定格式，每次复制粘贴）

```bash
cd docs && \
  cp state.md state.md.bak && \
  cp interfaces.md interfaces.md.bak && \
  cp decisions.md decisions.md.bak && \
  cp roadmap.md roadmap.md.bak && \
  echo "backups ok"
```

### 备份清理时机

- **不得**在当次任务结束时删除 `.bak`。
- 在**下一次**任务开始、创建新备份时自动覆盖旧 `.bak`（即上面的命令会自动覆盖，无需手动清理）。
- 始终保持 rollback depth = 1。

## 五、切片开发标准流程（每个切片的完整执行序列）

```
[接收任务]
    ↓
1. 读取权威文档
   必读：state.md, interfaces.md
   选读：decisions.md（了解历史约束）
    ↓
2. 检查接口变更
   IF 代码实现需要改变 interfaces.md 中的 API shape
   THEN 输出接口变更提案 → 停止 → 等待用户确认
   ELSE 继续
    ↓
3. 备份 4 个文档（覆盖旧 .bak）
    ↓
4. 输出实现计划（<= 6 步，先写后执行）
    ↓
5. 逐文件写代码（优先给完整文件内容）
   规则：
   - 最小改动，不做任何计划外的重构
   - 不引入未来切片的功能模块
   - 不修改禁止修改的文件列表之外的文件
    ↓
6. 输出验收步骤（命令 + 期望输出）
    ↓
7. 更新文档（按第三章触发规则）
   state.md：Slice 状态改为 "code written; pending run-verification"
   decisions.md：追加本切片的新决策
   roadmap.md：当前切片标注 "code written YYYY-MM-DD"
    ↓
[等待用户反馈]
    ↓
IF 用户说"测试通过"或终端日志显示预期输出
THEN:
  - 再次备份（新轮换）
  - state.md：Acceptance Tests 打 [x]，Slice 状态改为 "DONE -> 下一切片 NOT STARTED"
  - roadmap.md：当前切片移入 Done 区，下一切片升为 Now
ELSE:
  - 定位问题，最小修复，重新验证
```

## 六、Non-Fabrication 规则（最高优先级约束）

以下情况禁止在文档中写具体值：

| 情况 | 错误写法 | 正确写法 |
|------|---------|---------|
| 运行时版本未确认 | `ollama 0.3.2` | `TBD (ollama / llama.cpp / vLLM)` |
| 测试未运行 | `- [x] /health 返回 ok` | `- [ ] /health 返回 ok` |
| 端口未确定 | `Port: 8000` | `Port: TBD` |
| 命令未验证 | `pip install -r requirements.txt` ← 已安装则可以写 | 未安装时写 `（待验证）` |

触发此规则时，在 state.md 对应位置补充 checklist 项：
```markdown
- [ ] TODO: 确认 LLM runtime 版本并更新 Tech choices
```

## 七、接口变更提案流程

```
[发现需要修改 interfaces.md]
    ↓
输出提案，格式如下：

---
接口变更提案
变更端点：POST /chat
变更字段：新增 request.params.stream: boolean (optional)
变更原因：<一句话>
变更后 schema：
{
  "message": "string",
  "params": {
    "temperature": "number (optional)",
    "max_tokens": "number (optional)",
    "stream": "boolean (optional)"   ← 新增
  }
}
---
    ↓
停止。等待用户确认。
    ↓
IF 用户确认
THEN 先更新 interfaces.md，再动代码
ELSE 放弃变更，用现有 schema 实现
```

## 八、硬约束清单（每次执行前对照检查）

```
[ ] 已读 state.md 和 interfaces.md
[ ] 未改变 interfaces.md 中的 API shape（除非已提案并获确认）
[ ] 未引入本切片 Non-goals 中列出的功能模块
[ ] 未修改禁止修改的文件
[ ] 备份 4 个文档
[ ] 输出包含：计划 + 代码 + 验收步骤 + 文档更新
[ ] 文档中没有发明的事实（未知项写 TBD）
[ ] Acceptance Tests 只在用户/终端确认后才打 [x]
```

## 十、文件命名与目录约定

```
backend/
  main.py              ← FastAPI app 入口，只含路由和中间件
  llm_provider.py      ← LLMProvider 实现，与 main.py 解耦
  requirements.txt     ← 固定格式：包名>=最低版本

frontend/
  index.html           ← Vite 入口，不修改
  vite.config.js       ← Vite 配置，指定 port=3000，不修改
  package.json         ← 依赖声明
  src/
    main.jsx           ← React 挂载点，不修改
    App.jsx            ← 唯一业务组件（S0/S1 阶段）

docs/
  state.md             ← 每切片必读
  interfaces.md        ← 改代码前必读
  decisions.md         ← 有选型决策时追加
  roadmap.md           ← 切片状态变化时更新
  docOps.md            ← 本手册的元规范
  *.md.bak             ← 备份文件，不要提交到 git
```
