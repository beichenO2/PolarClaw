# 1. 设计目标

本系统是一个分层的、可扩展的、多工作模式的执行系统。  
系统必须同时满足以下要求：

- 支持多类任务，不预设单一主流程
    
- 支持不同任务类型拥有不同主状态机
    
- 支持共享工作流复用
    
- 支持局部经验沉淀与复用
    
- 支持未来自进化，但不允许自进化直接破坏系统稳定性
    
- 顶层对用户输入保持高泛化能力
    
- 下层执行接口保持强结构化、可验证、可追踪
    

因此，系统采用：

> **顶层 Markdown 语义控制 + 下层 JSON 结构化执行 + 分层状态机体系**

本设计采用以下对内统一命名。

## 2.1 顶层对象

- **CLAW**  
    系统最外层编排与治理实体
    
- **Mode**  
    工作模式定义
    
- **Executor**  
    某个 Mode 下的执行器
    

## 2.2 流程对象

- **Main Workflow FSM**  
    某个 Mode 的主状态机
    
- **Shared Workflow FSM**  
    可跨 Mode 复用的共享状态机
    
- **Run**  
    一次具体执行实例
    
- **Agent**  
    Run 内的执行单元
    
- **Pattern**  
    可复用的微型过程单元
    
- **Skill**  
    具体工具能力或外部动作能力

# 3. 系统分层

系统分为六层。

## Layer 0: User / Web UI

用户交互层。  
负责收集输入、展示系统状态、展示执行痕迹、接收人工确认。

## Layer 1: CLAW

系统总控层。  
负责需求理解、Mode 审视、执行路径审视、任务标准化、总控治理。

## Layer 2: Mode

工作模式层。  
负责定义一类任务的目标范式、适用范围、主执行逻辑集合。

## Layer 3: Executor

Mode 的执行层。  
负责在某个 Mode 下组织具体执行。

## Layer 4: Workflow FSM

流程控制层。  
负责用状态机管理一类执行流程。

## Layer 5: Run / Agent / Pattern / Skill

实际执行层。  
负责实例化运行、调用能力、沉淀局部经验。

# 4. 顶层架构定义

## 4.1 CLAW

### 定义

CLAW 是系统的最外层控制实体。  
CLAW 不是具体任务执行器，而是**总控、治理与审视层**。

### 职责

CLAW 负责：

1. 接收用户自然语言输入
    
2. 基于人类可读的 Markdown 标准理解需求
    
3. 对候选 Mode 进行审视
    
4. 对执行路径进行审视
    
5. 判断当前系统状态是否允许该执行路径
    
6. 输出标准化任务定义
    
7. 将任务交给对应 Executor
    

### 关键点

CLAW 的职责**不是简单选择 Mode**，而是：

- **定义与解释当前任务应落入何种 Mode**
    
- **审视该 Mode 是否合理**
    
- **审视当前系统状态是否支持该 Mode 的执行**
    
- **审视当前执行路径是否存在冲突、缺失或风险**
    

因此，CLAW 是：

- **治理者**
    
- **审查者**
    
- **任务标准化器**
    
- **顶层编排者**
    

而不是普通路由器。

---

## 4.2 Mode

### 定义

Mode 是一类任务的工作范式定义。

### 作用

Mode 决定：

- 该类任务的目标类型
    
- 该类任务允许的输出形式
    
- 该类任务使用哪些 Main Workflow FSM
    
- 该类任务默认启用哪些 Shared Workflow FSM
    
- 该类任务常用哪些 Pattern 与 Skill
    
- 该类任务的评估标准
    

### 说明

Mode 是一个**定义对象**，不是运行实例。

### 当前最小集合

- `knowledge_mode`
    
- `project_mode`
    

未来可扩展：

- `research_mode`
    
- `writing_mode`
    
- `debug_mode`
    
- `analysis_mode`
    

---

## 4.3 Executor

### 定义

Executor 是某个 Mode 下的实际执行主体。

### 作用

Executor 接收来自 CLAW 的标准化任务，并在该 Mode 的约束下组织执行。

### 职责

- 选择 Main Workflow FSM
    
- 组织 Shared Workflow FSM
    
- 创建 Run
    
- 调度 Agent
    
- 调用 Pattern / Skill
    
- 收集结果
    
- 回传给 CLAW
    

### 示例

- `knowledge_executor`
    
- `project_executor`
    

---

# 5. 状态机体系定义

状态机不再定义为“一个全局统一状态机”，而是定义为一个层次化体系。

---

## 5.1 Main Workflow FSM

### 定义

Main Workflow FSM 是某个 Mode 的主执行状态机。

### 作用

它定义该类任务的主流程骨架。

### 特点

- 由某个 Executor 选用
    
- 一次任务通常绑定一个 Main Workflow FSM
    
- 管理该类任务的主生命周期
    
- 可以调用 Shared Workflow FSM
    
- 可以在状态中创建 Run
    

### 示例

- `KnowledgeExplainFSM`
    
- `ProblemSolvingFSM`
    
- `ProjectMainFSM`
    
- `ProjectDebugFSM`
    

---

## 5.2 Shared Workflow FSM

### 定义

Shared Workflow FSM 是可跨 Mode 复用的共享流程状态机。

### 作用

处理跨任务复用的制度化流程。

### 特点

- 可被多个 Main Workflow FSM 调用
    
- 负责治理性流程，而非领域主流程
    
- 通常与验证、人工介入、补录、审查有关
    

### 示例

- `HumanActionFSM`
    
- `ManualRunReconcileFSM`
    
- `ValidationFSM`
    
- `ReviewFSM`
    

---

## 5.3 Pattern

### 定义

Pattern 是可复用的微型过程单元。

### 作用

沉淀局部经验，提高执行质量与稳定性。

### 特点

- 小
    
- 具体
    
- 高复用
    
- 可跨 Mode
    
- 不一定是状态机
    

### Pattern 的合法形式

Pattern 可以是：

- checklist
    
- mini-FSM
    
- recipe
    
- decision-tree
    
- rule-pack
    

### 示例

- `APISetupPattern`
    
- `DebugTriagePattern`
    
- `KnowledgeChainPattern`
    
- `LayeredExplanationPattern`
    

---

# 6. Run 执行模型

## 6.1 Run

### 定义

Run 是一次具体执行实例。

### 性质

- 有唯一 `run_id`
    
- 有状态
    
- 有输入
    
- 有证据
    
- 有日志
    
- 有结果
    
- 可追踪
    
- 可回放
    

### Run 的来源

Run 由 Workflow FSM 在合适状态发起。

### Run 的职责

- 绑定一次具体执行
    
- 管理 Agent 生命周期
    
- 收集 Evidence
    
- 提交 Validation
    
- 产出 Result
    

---

## 6.2 Agent

### 定义

Agent 是 Run 内的执行单元。

### 职责

- 承接某个子任务
    
- 调用 Skill
    
- 应用 Pattern
    
- 输出结果与证据
    

### 特点

- 可以并发
    
- 可以专业化
    
- 可以使用不同模型或配置
    

---

## 6.3 Skill

### 定义

Skill 是最具体的能力单元。

### 类型

- 工具调用
    
- 本地命令
    
- 文件操作
    
- 网络探测
    
- 文档更新
    
- 验证器执行
    

### 关系

Agent 通过 Skill 执行具体动作。  
Pattern 可以组织 Skill 的使用顺序。

# 7. Markdown 与 JSON 的控制面划分

这是本系统的重要设计。

---

## 7.1 顶层使用 Markdown

### 范围

CLAW 层使用 Markdown。

### 原因

顶层需要处理：

- 人类自然语言输入
    
- 高泛化需求理解
    
- 抽象意图解析
    
- Mode 审视
    
- 执行路径审视
    
- 系统状态反思
    

这些任务天然适合使用人类可读的、语义密度高的文本控制格式。

### CLAW 的输入输出

CLAW 读取：

- 用户输入
    
- 系统规范文档
    
- Mode 定义文档
    
- 当前系统状态摘要
    
- 当前 Executor / FSM 能力摘要
    

CLAW 输出：

- 一个规范化的任务分析 Markdown
    
- 一个任务决策结果
    
- 一个结构化任务派发对象
    

---

## 7.2 下层使用 JSON

### 范围

从 Executor 开始往下，全部使用 JSON 或等价强结构格式。

包括：

- Mode 实例化输入
    
- Executor 任务对象
    
- FSM 定义
    
- Run 定义
    
- AgentTask
    
- Pattern 元数据
    
- Skill 调用参数
    
- ValidationReport
    
- EvidenceIndex
    

### 原因

下层必须：

- 强约束
    
- 可验证
    
- 可自动解析
    
- 可追踪
    
- 可回放
    
- 可版本管理
    

因此不应继续使用自由文本作为执行控制面。

# 8. 顶层审视机制：CLAW 如何审视 Mode 与执行路径

你新增的这一点是本设计的关键部分。

CLAW 不只是接收用户输入并“选一个 Mode”，而是要承担**顶层审视功能**。

---

## 8.1 审视对象

CLAW 至少要审视以下三类对象：

1. **Mode 定义是否合理**
    
2. **当前任务进入该 Mode 是否合理**
    
3. **当前系统状态下，该执行路径是否合理**
    

---

## 8.2 审视结果类型

CLAW 对 Mode 与执行路径的审视，应输出以下结果之一：

- `accepted`  
    通过，可执行
    
- `accepted_with_warnings`  
    可以执行，但存在警告
    
- `rejected`  
    不通过，需要改 Mode 或改执行路径
    
- `needs_revision`  
    需要修订 Mode 定义或任务映射
    

---

## 8.3 审视维度

CLAW 的审视应覆盖：

### A. 任务匹配性

当前用户需求是否适合该 Mode

### B. 能力完备性

当前 Mode/Executor/FSM/Pattern/Skill 组合是否足以处理任务

### C. 执行路径完整性

是否缺失关键步骤、验证点、等待点或回退点

### D. 系统状态一致性

当前系统配置、文档、注册表、资源状态是否支持该路径

### E. 自进化兼容性

该路径是否符合未来可演进原则，是否会造成不可控漂移

# 9. Web UI 交互设计

---

## 9.1 顶层 UI

Web UI 顶层直接与 CLAW 交互。

### 顶层输入

- 用户自然语言需求
    
- 用户可选的 Mode 建议
    
- 用户可选项目上下文
    
- 用户可选约束
    

### 顶层输出

- CLAW 的任务分析
    
- 推荐 Mode
    
- 审视结果
    
- 执行建议
    
- 确认后进入对应 Executor 工作台
    

---

## 9.2 UI 流程

### Step 1: 用户提交需求

用户输入自然语言任务。

### Step 2: CLAW 解析并审视

CLAW 基于 Markdown 规范输出：

- 任务理解
    
- 候选 Mode
    
- Mode 审视结论
    
- 执行路径审视结论
    
- 推荐 Executor
    

### Step 3: 用户确认

用户确认进入某个 Mode / Executor。

### Step 4: 创建标准化任务对象

CLAW 生成 JSON 任务对象并派发给 Executor。

### Step 5: Executor 工作台

进入相应 Executor 的工作台界面。

---

## 9.3 Executor 工作台

每个 Executor 工作台至少展示：

- 当前任务摘要
    
- Main Workflow FSM 状态
    
- Shared Workflow FSM 列表
    
- Run 列表
    
- Agent 监控
    
- Pattern 使用记录
    
- Evidence / Validation 面板
    
- Human Action / Manual Reconcile 面板（如适用）

# 13. 自进化兼容策略

系统必须兼容未来自进化，但自进化不得直接修改正式执行路径。

因此定义如下规则。

## 13.1 CLAW 可审视的对象

CLAW 可以审视：

- Mode 定义
    
- Executor 配置
    
- FSM 结构
    
- Pattern 适用性
    
- 当前执行路径
    

## 13.2 自进化输出形式

自进化输出不得直接生效。  
必须以以下形式产出：

- Mode Revision Proposal
    
- FSM Revision Proposal
    
- Pattern Proposal
    
- Executor Capability Proposal
    

## 13.3 生效机制

所有自进化改动必须经过：

1. 结构检查
    
2. 兼容性检查
    
3. 审视层批准
    
4. 注册后才能进入正式运行域

# 15. 最终定义

本系统的唯一推荐定义如下：

> 本系统是一个由 CLAW 顶层治理、以 Mode 为任务范式、由 Executor 承担模式执行、以 Main Workflow FSM 与 Shared Workflow FSM 管理流程、以 Run 实例化执行、以 Agent 执行任务、以 Pattern 沉淀经验、以 Skill 实现具体能力的分层执行系统。

其核心原则是：

1. 顶层使用 Markdown 处理高泛化语义与治理审视
    
2. 下层使用 JSON 处理强结构化执行
    
3. Mode 不是简单选择器，而是工作范式定义
    
4. CLAW 不只是路由器，而是 Mode 与执行路径的审视者
    
5. FSM 管理中层流程，Pattern 管理局部经验
    
6. 自进化只能通过提案进入系统，不能直接改写正式执行路径

前端（Vite + React，端口 3000）

单组件 `App.jsx`，包含两个功能区：

- 健康状态：每 5 秒轮询 `/health`，三态显示（checking 灰 / ok 绿 / failed 红），后端宕机后自动变红
- 聊天区：输入框 + Send 按钮，支持 Enter 发送，发送中 loading 状态，返回内容同时显示 model 名和 latency

---

### 二、文档体系（项目的外部记忆）

这是整个项目里含金量最高的部分。我们建立了一套 5 文件文档系统，让 Agent 跨对话不失忆：

|文件|角色|核心规则|
|---|---|---|
|`state.md`|SSOT，最高权威|冲突时其他文档向它对齐|
|`interfaces.md`|API 契约|代码必须跟它一致，改接口必须先改它|
|`decisions.md`|决策日志|只能追加，永不改写历史，共记录了 D001–D009|
|`roadmap.md`|Backlog|Done/Now/Next/Later 四区，切片推进时迁移|
|`docOps.md`|操作规范本身|什么时候改哪个文档、怎么备份、怎么回滚|

`state.md` 实时记录了切片状态（S0 DONE → S1 DONE → S2 NOT STARTED）、tech choices、运行命令、验收 checklist，每次验证通过才打 `[x]`。

---

### 三、工程流程规范

我们在执行中形成并固化了几个重要规范：

切片开发（Vertical Slice）：每个切片有明确的 Non-goals（不做什么），先实现最小闭环，再推进下一切片，不允许跨切片引入功能。

DocOps + 备份轮换：每次修改文档前必须先创建 `.bak`，保持 rollback depth = 1。旧备份在下次任务开始时自然被覆盖，不手动清理。

接口变更门控：发现需要改 `interfaces.md` 的 schema → 停下 → 先提案 → 等确认 → 才能动代码。整个 S0/S1 过程中接口从未被破坏性修改。

Non-Fabrication：文档中未确认的事实写 TBD，不猜测，不提前打 `[x]`。

---

### 四、Agent 调度层（`prompt` 文件）

根目录的 `prompt` 文件是一个可复用的 Agent 任务模板，定义了受控工程实现 Agent 的调用格式：硬约束、允许/禁止修改的文件列表、上下文注入点、输出格式要求。每次发起新切片任务时套这个模板填写，而不是自由发挥。

---

### 五、知识沉淀（两份经验文档）

- `经验.md`：以本项目为具体案例，记录了每个文档的实际格式、S0/S1 的执行轨迹、文件改动数量
- `通用经验.md`：抽象为技术栈无关的通用 Agent 工程手册，包含完整模板（占位符版）、触发规则表、执行流程图、常见错误对照表，可以直接用于下一个完全不同的项目

---

### 重点/亮点总结

1. 从零到双切片可运行系统，文档与代码同步演进，每个阶段都有明确的 Acceptance Tests 且已验证
2. LLMProvider 抽象层：S1 用 EchoProvider 跑通了完整数据流，换真实 LLM 时不需要改路由层
3. 文档即合同（Contract-first）：interfaces.md 在写任何代码之前就定义好，全程未被违反
4. DocOps 是系统性工程，不只是"写文档"，而是规定了备份时机、更新触发条件、回滚方式、Non-Fabrication 约束，让文档系统可信
5. 可复用的 prompt 模板：把怎么给 Agent 下任务本身也做成了规范，降低了下次启动新切片的沟通成本
6. 通用经验.md 是可迁移的资产：和具体项目完全解耦，可以直接作为任何新项目的工程底座
   
   ## 上传到 GitHub 的完整流程

### 1. 前置检查

检查 GitHub CLI 登录状态：

gh auth status

确认已登录账号、有 `repo` 权限、以及 git 操作协议（ssh/https）。若未登录则需先执行 `gh auth login`。

---

### 2. 初始化本地仓库并提交

git init

git add .

git commit -m "Initial commit: add project files"

---

### 3. 一条命令创建远程仓库并推送

gh repo create PolarClaw --public --source=. --remote=origin --push

关键参数说明：

|参数|作用|
|---|---|
|`PolarClaw`|仓库名（取的是项目文件夹名）|
|`--public`|创建公开仓库（也可用 `--private`）|
|`--source=.`|以当前目录作为本地源|
|`--remote=origin`|自动绑定远程为 `origin`|
|`--push`|创建后立即推送，无需额外 `git push`|

---

### 关于"如何确认没有重名项目"

实际上我没有提前检查，而是依赖 `gh repo create` 的行为：

- 若名称已被自己占用 → 命令报错，不会覆盖已有仓库
- 若名称已被他人占用 → 不影响，GitHub 仓库名是按账号隔离的（`beichenO2/PolarClaw`）

如果想主动检查，可以用：

gh repo view beichenO2/PolarClaw 2>&1 | grep -q "Could not resolve" && echo "不存在" || echo "已存在"

---

### 为什么全程无需用户干预

核心是 GitHub CLI（`gh`） 的能力：

- 认证信息已通过 `gh auth login` 提前存入系统 keyring，无需每次输入密码
- `gh repo create --source --push` 将"建仓库 + 关联远程 + 推送"合并为一步
- 使用 SSH 协议，推送时也不需要输入凭据
