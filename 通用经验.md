# 通用 Agent 工程手册

适用范围：任何软件项目，任何技术栈。
使用方式：接到任务时先读第一章定向，再按对应章节执行，不需要通读。

---

## 第一章 项目记忆体系

Agent 没有跨对话记忆。文档是唯一的外部记忆。
每个项目必须在 `docs/` 目录下维护以下 5 个文件：

| 文件 | 职责 | 可信度 |
|------|------|--------|
| `state.md` | 项目当前状态的单一事实来源（SSOT） | 最高，其他文档与之冲突时以此为准 |
| `interfaces.md` | API / 接口契约，代码必须与之一致 | 高，改代码前必须先改此文件 |
| `decisions.md` | 历史决策日志，只能追加 | 中，帮助理解约束来源 |
| `roadmap.md` | 待办 Backlog，允许过时 | 低，仅供参考，不作为权威 |
| `docOps.md` | 文档操作规范（即本手册的项目内副本） | 执行规范，不记录业务信息 |

**冲突解决顺序：**

```
state.md > interfaces.md > decisions.md > roadmap.md
```

发现文档间冲突时：以 state.md 为准，同时输出 patch 修正其他文档，不要"取平均"。

## 第二章 文档模板

### 2.1 state.md 模板

```markdown
# Project State (SSOT)

Last Updated: YYYY-MM-DD
Source of Truth: state.md is authoritative.
Conflict Rule:
- If other docs conflict with state.md, follow state.md.
- Propose patches to bring other docs in sync. Do not "average" conflicts.

## Project
Name: <项目名>
Goal (v1): <一句话目标>

## Current Iteration
Iteration: <IN-N STATUS -> IN-N+1 STATUS>

- IN-0: <描述> [状态标签]
- IN-1: <描述> [状态标签]

Non-goals (for current iteration):
- No <禁止做的事>

## Architecture
Layers:
- <层名>: <职责>
- <层名>: <职责>

Data flow:
<组件A> -> <接口> -> <组件B> -> <响应> -> <组件A>

## Tech Stack
<组件>: <选型，未确定写 TBD (候选A / 候选B)>
<组件>: <选型>

## Repo Layout
- <目录>/   (<实际包含的文件说明>)
- <目录>/

## Run Targets
<服务名>:
- Port: <端口>
- Command: <完整可执行命令>

## Acceptance Tests
IN-0:
- [ ] <验收项>
IN-1:
- [ ] <验收项>
```

**state.md 字段规则：**

```
Last Updated      → 每次修改时更新，可附注原因
Iteration 行      → 格式固定："IN-N STATUS -> IN-N+1 STATUS"
状态标签           → 只用以下四种之一：
                    [NOT STARTED]
                    [code written YYYY-MM-DD; pending verification]
                    [DONE - verified YYYY-MM-DD]
                    [IN PROGRESS]
Tech Stack        → 未确定的写 TBD，确定后更新，不得凭猜测填写
Run Targets       → 写完整命令，可直接复制粘贴执行
Acceptance Tests  → 只在用户或终端输出确认通过后才标 [x]
```

### 2.2 interfaces.md 模板

```markdown
# Interfaces (Contract)

Rule:
- Treat this file as the contract.
- Do NOT change interface shapes in code unless this file is updated first.

## <接口分组，如 HTTP API / gRPC / CLI>

### <方法> <路径或名称>
Purpose: <一句话，标注适用迭代>

Request:
<请求结构，含字段名、类型、是否可选>

Response <状态码>:
<响应结构>

Error Response:
<统一错误结构>

## Internal Interfaces

### <内部接口名>
- <方法签名>
Notes:
- <约束>

(Reserved for future)
### <未来接口名> (IN-N+)
- <方法签名>
```

**interfaces.md 字段规则：**

```
字段类型标注     → "string" / "number" / "boolean" / "string (optional)"
Error 结构       → 全项目统一，所有端点共用同一 error schema
未来接口          → 只写签名，不写实现说明，标注 (IN-N+)
修改前置条件      → 必须先提案，等用户确认，再动代码
```

### 2.3 decisions.md 模板

```markdown
# Decisions Log (Append-only)

Rules:
- Append new entries only. Do not rewrite old entries.
- Each decision should be small, explicit, and reversible where possible.

## YYYY-MM-DD
- D001: <做了什么选择>（<一句话理由，不超过20字>）。
- D002: <做了什么选择>（<理由>）。

## YYYY-MM-DD (IN-N)
- D00N: <做了什么选择>（<理由>）。
```

**decisions.md 填写规则：**

```
只追加，绝不修改已有行。
D-ID 全局唯一，从 D001 顺序递增。
触发条件（以下情况必须追加）：
  - 技术框架 / 运行时 / 语言选型
  - 端口 / 协议 / 传输方式确定
  - 数据存储方案选定
  - API 设计取舍（为何选 A 不选 B）
  - 安全策略（认证方式、跨域策略等）
不触发条件（以下情况不追加）：
  - 变量命名
  - 代码格式调整
  - 注释修改
```

### 2.4 roadmap.md 模板

```markdown
# Roadmap / Backlog

## Done
- IN-0: <描述> [verified YYYY-MM-DD]

## Now
- IN-1: <当前迭代描述>

## Next
- IN-2: <描述>
- IN-3: <描述>

## Later / Parking Lot
- <无明确排期的长期需求>
```

**roadmap.md 维护规则：**

```
Done 区   → 迭代验收通过后移入，标注 verified 日期
Now 区    → 只有一项，即当前正在进行的迭代
Next 区   → 按优先级排列，不含 Done/Now 中已有的条目
Later 区  → 无排期的愿望清单，可长期停留
移动规则  → 同一条目不能同时出现在两个区，移入新区时从旧区删除
```

### 2.5 docOps.md 模板

```markdown
# DocOps Policy

## Managed Documents
- docs/state.md
- docs/interfaces.md
- docs/decisions.md
- docs/roadmap.md

## Authority Order
state.md > interfaces.md > decisions.md > roadmap.md

## Update Triggers
（粘贴本手册第三章内容）

## Backup & Rotation
（粘贴本手册第四章内容）

## Non-Fabrication Rule
The agent MUST NOT invent facts in docs.
Unknown values → write "TBD" + add a [ ] checklist item in state.md.

## Per-Task Output Requirements
1) Plan (<= 6 steps)
2) Code changes (full file content preferred)
3) Verification steps (commands + expected output)
4) Doc patches (if triggered)
```

## 第三章 文档更新触发规则

每轮任务结束前，对照此表判断需要更新哪些文档：

| 发生的事件 | 必须更新 |
|-----------|---------|
| 迭代状态变化（开始 / 代码写完 / 验收通过） | state.md |
| 技术选型从 TBD 变为确定 | state.md + decisions.md |
| 启动命令 / 端口变化 | state.md |
| 新增 / 删除 / 修改接口端点或字段 | interfaces.md（需先提案）|
| 验收测试通过（用户确认或终端输出验证） | state.md（打 `[x]`）|
| 有框架 / 协议 / 安全策略决策 | decisions.md |
| 迭代验收完成，进入下一迭代 | state.md + roadmap.md |
| 发现新功能需求 | roadmap.md |

**不触发更新的事件（不要误操作）：**

```
- 代码内部重构（不影响对外行为）
- 变量 / 函数重命名
- 测试文件内容变化
- 注释修改
- 未经验证的"应该能工作"
```

## 第四章 备份与轮换

### 规则

```
修改任意文档之前：必须先备份全部 4 个文档（覆盖旧 .bak）。
任务结束时：不得删除 .bak。
下次任务开始时：执行备份命令会自动覆盖旧 .bak，即完成轮换。
结果：始终保持 rollback depth = 1。
```

### 固定命令（每次直接执行）

```bash
cd docs && \
  cp state.md state.md.bak && \
  cp interfaces.md interfaces.md.bak && \
  cp decisions.md decisions.md.bak && \
  cp roadmap.md roadmap.md.bak && \
  echo "backup ok"
```

### 回滚方式（需要时）

```bash
cd docs && \
  cp state.md.bak state.md && \
  cp interfaces.md.bak interfaces.md && \
  cp decisions.md.bak decisions.md && \
  cp roadmap.md.bak roadmap.md && \
  echo "rollback ok"
```

## 第五章 每轮任务的执行序列

```
[接收任务]
    │
    ▼
① 读取上下文
   必读：state.md, interfaces.md
   选读：decisions.md（了解历史约束）
   目的：确认当前迭代、当前 Non-goals、现有接口形状
    │
    ▼
② 判断接口是否需要变更
   IF 实现需要改变 interfaces.md 中的任何 schema/签名
   THEN → 执行第六章【接口变更提案流程】，停止等待确认
   ELSE → 继续
    │
    ▼
③ 备份 4 个文档（第四章命令）
    │
    ▼
④ 输出实现计划（<= 6 步）
   格式：
     步骤1：<动词> <文件/组件>
     步骤2：...
    │
    ▼
⑤ 执行代码变更
   规则：
   - 最小改动，不重构计划外的代码
   - 不引入当前迭代 Non-goals 中的功能
   - 不修改明确禁止修改的文件
   - 优先给完整文件内容，而非 diff 片段
    │
    ▼
⑥ 输出验收步骤
   格式：
     命令：<可直接执行的命令>
     期望输出：<具体的期望值，不写"应该成功"这类模糊描述>
    │
    ▼
⑦ 更新文档（按第三章触发规则）
   state.md → 迭代状态改为 "code written; pending verification"
   decisions.md → 追加本轮新决策
   roadmap.md → 当前迭代标注 "code written YYYY-MM-DD"
    │
    ▼
[等待用户反馈]
    │
    ├─ 用户确认测试通过
    │       │
    │       ▼
    │   再次备份（新轮换）
    │   state.md：Acceptance Tests 打 [x]
    │   state.md：Slice 状态 → "DONE -> 下一迭代 NOT STARTED"
    │   roadmap.md：当前迭代移入 Done，下一迭代升为 Now
    │
    └─ 用户报告问题
            │
            ▼
        定位最小复现路径
        最小修复，不引入新变更
        重新输出验收步骤
        循环回到[等待用户反馈]
```

## 第六章 接口变更提案流程

```
[发现需要修改 interfaces.md]
    │
    ▼
输出提案，必须包含：
  - 变更的接口名称
  - 变更前的 schema（原文引用）
  - 变更后的 schema（完整新版）
  - 变更原因（一句话）
    │
    ▼
停止执行。等待用户确认。不得在确认前修改任何代码。
    │
    ├─ 用户确认
    │       │
    │       ▼
    │   先更新 interfaces.md → 然后写代码
    │
    └─ 用户拒绝
            │
            ▼
        在现有 schema 约束内寻找实现方案
```

**提案输出格式：**

```
---
接口变更提案
变更接口：<接口名>
变更类型：新增字段 / 删除字段 / 修改字段类型 / 新增端点 / 删除端点
变更前：
  <原 schema>
变更后：
  <新 schema>
原因：<一句话>
---
等待确认后继续。
```

## 第七章 Non-Fabrication 规则

**禁止在文档中写入未经确认的事实。**

| 情况 | 错误写法 | 正确写法 |
|------|---------|---------|
| 版本未确认 | `redis 7.2.4` | `TBD (redis / valkey)` |
| 测试未运行 | `- [x] 接口返回 200` | `- [ ] 接口返回 200` |
| 命令未验证 | `npm run build` | `npm run build（待验证）` |
| 端口未决定 | `Port: 3000` | `Port: TBD` |
| 性能未测量 | `latency < 100ms` | `latency: TBD` |

写入 TBD 时，同步在 state.md 的 Acceptance Tests 或末尾添加：

```markdown
- [ ] TODO: 确认 <具体项目> 并更新文档
```

## 第八章 硬约束清单

每次开始任务前对照执行，全部为 true 才继续：

```
[ ] 已读 state.md，知道当前迭代和 Non-goals
[ ] 已读 interfaces.md，知道当前所有接口形状
[ ] 未计划修改 interfaces.md 中的 schema（或已提案并获确认）
[ ] 未计划引入当前迭代 Non-goals 中列出的功能
[ ] 未计划修改被明确标注为禁止修改的文件
[ ] 备份 4 个文档已执行
[ ] 输出包含：计划 + 代码 + 验收步骤 + 文档更新
[ ] 文档中没有发明的事实（未知项写 TBD）
[ ] Acceptance Tests 只在确认后才标 [x]
```

## 第九章 项目初始化（从零开始时执行一次）

```
步骤1：创建目录结构
  mkdir -p docs <src目录按项目决定>

步骤2：创建 docs/docOps.md
  内容：粘贴本手册第二章 2.5 节模板，填入项目专属内容

步骤3：创建 docs/state.md
  内容：填写 2.1 节模板
  - Project.Name 和 Goal 必须填
  - Tech Stack 未决定的写 TBD
  - Acceptance Tests 全部写 [ ]
  - Iteration 状态写 IN-0 NOT STARTED

步骤4：创建 docs/interfaces.md
  内容：填写 2.2 节模板
  - 至少写第一个接口（如 /health）
  - 未来接口占位写签名 + (IN-N+) 标注

步骤5：创建 docs/decisions.md
  内容：填写 2.3 节模板
  - 首批决策通常包含：
    D001: 部署方式选择
    D002: 迭代策略（垂直切片 / 水平分层等）
    D003: 入口方式（CLI / WebUI / API 等）
    D004: 接口优先策略

步骤6：创建 docs/roadmap.md
  内容：填写 2.4 节模板
  - Now 区写 IN-0
  - Next 区写 IN-1 到 IN-N（已知需求）
  - 不确定的放 Later

步骤7：执行第一次备份（第四章命令）
```

## 第十章 常见错误与对应处理

| 错误行为 | 正确处理 |
|---------|---------|
| 看到 TBD 就随意填一个值 | 保持 TBD，将确认动作写为 checklist 项 |
| 发现接口需要改，直接改代码 | 先输出变更提案，等确认，再改 interfaces.md，最后改代码 |
| 任务完成后删除 .bak | 不删除，等下次任务开始时自动覆盖 |
| 在文档中提前打 [x] | 只有终端输出或用户话语明确确认后才打 [x] |
| 顺手把下个迭代的功能一起写了 | 删除，当前迭代 Non-goals 是硬约束 |
| 多个文档数据不一致，取"中间值" | 以 state.md 为准，修正其他文档至与之一致 |
| 文档没有备份就直接修改 | 停下，先执行备份命令，再继续 |
| 给出模糊验收步骤（"应该没问题"） | 必须给出具体命令和期望的具体输出值 |
